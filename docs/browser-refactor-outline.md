# 浏览器适配层重构大纲

## 1. 目标
- 将浏览器相关代码、资源和启动参数收敛到清晰的目录边界内。
- 降低主流程与浏览器实现之间的耦合，方便后续迁移到其他项目。
- 统一资源加载入口，避免同类逻辑散落在多个位置。
- 保持现有行为兼容，优先做结构整理，再做内部拆分。

## 2. 现状问题
- 浏览器启动参数、上下文初始化、页面脚本加载逻辑分散在多个文件。
- 扩展资源、注入脚本、辅助脚本、启动桥接文件的路径规则不统一。
- 部分逻辑存在重复实现，后续维护时容易出现修改遗漏。
- 资源打包和运行时读取路径存在多套兜底规则，迁移成本较高。

## 3. 重构原则
- 先归类，再拆分，不改变对外接口优先。
- 统一命名和目录职责，减少跨目录引用。
- 让“配置”“脚本”“资源”“桥接”各自独立。
- 保持可回退，迁移时不一次性大改所有调用点。

## 4. 建议目录结构
- `src/core/browser/config/`：浏览器启动与运行时配置。
- `src/core/browser/profile/`：浏览器配置生成、区域与语言参数。
- `src/core/browser/runtime/`：浏览器生命周期、上下文管理、页面绑定。
- `src/core/browser/resources/`：浏览器相关脚本、模板、注入资源，当前已建立。
- `src/core/browser/resources/stealth/`：伪装与注入相关资源的统一子目录，当前已建立。
- `src/core/browser/bridge/`：Electron 与 Playwright 的桥接代码。
- `extensions/<extension-name>/`：独立扩展目录，按扩展自身职责组织。

## 5. 资源统一方案
- 将浏览器相关静态脚本集中到单一资源目录。
- 将扩展内的页面脚本、后台脚本、辅助脚本按功能分层。
- 统一所有资源的解析入口，避免散落的 `path.join` 规则。
- 在打包配置里只保留一个资源根入口，减少重复拷贝逻辑。

## 6. 分步实施
### 第一步：盘点与标注
- 列出所有浏览器相关文件。
- 标注每个文件的职责、入口、依赖方。
- 找出重复逻辑和高耦合点。

### 第二步：建立统一资源入口
- 新增单一资源目录。
- 迁移可独立的脚本和模板到该目录。
- 保留旧路径的兼容读取，避免一次性破坏现有流程。

### 第三步：拆分启动与运行时逻辑
- 将启动参数构建、上下文初始化、页面附加逻辑分开。
- 让主流程只依赖少量稳定接口。
- 把与扩展无关的浏览器能力抽成公共模块。

### 第四步：整理扩展边界
- 每个扩展只保留自身必要的入口文件。
- 将共享工具抽到公共目录。
- 统一 manifest、资源路径和打包规则。

### 第五步：更新打包与迁移说明
- 补齐资源拷贝规则。
- 补一份迁移说明，记录旧路径到新路径的映射。
- 增加基础验证步骤，确保迁移后可以正常启动。

## 7. 验证清单
- 启动流程是否正常。
- 资源路径在开发态和打包态是否一致。
- 扩展是否能正确加载。
- 旧路径兼容是否仍然可用。
- 日志中是否还存在重复初始化或重复加载。

## 8. 迁移产物
- 统一后的资源目录。
- 更新后的路径解析工具。
- 更新后的打包配置。
- 迁移说明文档。
